# 03. 探索→計画→コーディングのアプローチに従う

*ソース: https://code.claude.com/docs/ja/best-practices*
*日付: 2026-04-02*

## 写経

# 最初に探索し、次に計画し、その後コーディングする

> 研究と計画を実装から分離して、間違った問題を解決することを避けます。

Claude が直接コーディングにジャンプさせると、間違った問題を解決するコードが生成される可能性があります。 Plan Mode を使用して、探索を実行から分離します。

推奨されるワークフローには4つのフェーズがあります。

### 1 探索

Plan Mode に入ります。 Claude はファイルを読み取り、変更を加えずに質問に答えます。

```claude (Plan Mode)
read /src/auth and understand how we handle sessions and login.
also look at how we manage environment variables for secrets.
```

### 2 計画

Claude に詳細な実装計画を作成するよう依頼します。

```claude (Plan Mode)
I want to add Google OAuth. What files need to change?
What's the session flow? Create a plan.
```

Ctrl+G を押して、 Claude が進む前に、テキストエディタで計画を開いて直接編集します。

### 3 実装

Normal Mode に戻り、 Claude にコーディングさせ、計画に対して検証します。

```claude (Normal Mode)
implement the OAuth flow from your plan. write tests for the callback handler, run the test suite and fix any failures.
```

### 4 コミット

Claude に説明的なメッセージでコミットし、PRを作成するよう依頼します。

```claude (Normal Mode)
commit with a descriptive message and open a PR
```

Plan Mode は便利ですが、オーバーヘッドも追加します。

> スコープが明確で修正が小さいタスク（タイプミスの修正、ログ行の追加、変数の名前変更など）の場合は、 Claude に直接実行するよう依頼します。
> 
> 計画は、アプローチについて不確実な場合、変更が複数のファイルを変更する場合、または変更されるコードに不慣れな場合に最も役立ちます。差分を一文で説明できる場合は、計画をスキップします。

## 気づき・疑問

### Q. 普段のClaude Code利用で、Plan Mode を使ったことはあるか？使った場合、どんな場面で有効だったか？

使ったことがある。場面は幅広い：
- 複数ファイルにまたがる変更（影響範囲の把握）
- 未知のコードベースの理解
- 設計判断の整理（アプローチの比較検討）
- 調査タスクの計画立案
- 問い合わせ対応やエラー調査

実装以外の「調査・分析」用途でも Plan Mode を活用しているのが特徴的。「安全に読み取り専用で探索する」ツールとして幅広く使えている。

### Q. 4つのフェーズ（探索→計画→実装→コミット）のうち、自分が飛ばしがちなフェーズはどれか？それによってどんな問題が起きたか？

特に飛ばしがちなフェーズはない。むしろ記事の想定（1セッション内でモード切替）を超えて、**フェーズごとにセッションを分離**している：

1. 探索セッション → 探索ドキュメント作成
2. 計画セッション → 計画ドキュメント・見積もり作成
3. 実装セッション → コーディング

各セッションで Plan Mode を活用。この方法のメリット：
- セッションごとにコンテキストがクリーンに保たれる
- ドキュメントが成果物として残り、次セッションに必要な情報だけ渡せる
- 見積もりまで計画フェーズに含めることで、実装の見通しが立つ

> **Claudeのコメント:** 記事は「1セッション内でPlan→Normal切替」を想定しているが、フェーズごとにセッションを分けてドキュメントを残すアプローチは、コンテキスト管理の観点でより優れている。セッション分離 + ドキュメント駆動は、記事のベストプラクティスを自然に発展させた形。

### Q. 「差分を1文で説明できる場合は、計画をスキップする」という基準について、自分の感覚と合っているか？境界線はどこにあると思うか？

感覚と合っている。小さなタスク（タイプミス修正、ログ追加、変数名変更など）では計画フェーズを入れずに直接実行している。

## 自分の仕事に使えそうなこと

- フェーズごとのセッション分離は既に実践できている。これを意識的なパターンとして定着させる
- 探索・計画フェーズのドキュメント化は強み。テンプレート化すればさらに効率的にできる可能性がある
- 調査・問い合わせ対応でもPlan Modeは使える。実装以外の用途も意識しておく
